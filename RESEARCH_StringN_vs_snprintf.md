# StringN vs snprintf

Короткое прикладное исследование для `TimberWidget`.

## Что сравниваем

Сравнение делается не между `Arduino String` и `StringN`, а между двумя способами собрать одну и ту же строку команды:

1. `StringN` через последовательные `add(...)`
2. `snprintf(...)` через форматную строку

Для практики это ближе всего к реальному вопросу: чем лучше собирать строки команд для телефона на микроконтроллере.

## Что важно заранее

`StringN` и `snprintf` имеют разный профиль сильных сторон:

- `StringN` работает со статическим буфером и не использует динамическую память.
- `StringN` использует собственные функции конвертации чисел и копирования строк.
- `snprintf` использует универсальный printf-движок с разбором форматной строки.
- Для очень простых строк один вызов `snprintf` может оказаться быстрым или близким по скорости.
- Для поэтапной сборки, особенно когда есть циклы по байтам, `StringN` часто оказывается выгоднее по контролю и обычно лучше предсказуем по памяти.

## Что говорит сам StringN

По README библиотеки `StringN`:

- библиотека позиционируется как "лёгкий и быстрый статический String-билдер"
- автор пишет, что она "в ~5 раз быстрее Arduino String"
- и "на ~3 кБ легче Arduino String"

Это не прямое сравнение с `snprintf`, но важный ориентир по замыслу библиотеки.

Отдельно в README указано:

- `STRN_DEFAULT_FLOAT` переключает float-конвертацию на `dtostrf`, это примерно на 20% быстрее, но тяжелее примерно на 1.2 кБ
- `STRN_DEFAULT_INT` переключает целые числа на `ltoa/ultoa`, и автор прямо помечает этот режим как "сильно медленнее"

## Что видно по исходникам

Для `StringN`:

- добавление строк делается через прямое копирование в буфер `memcpy`
- длина хранится отдельно в `_len`, поэтому не нужен постоянный `strlen` уже готовой строки
- числа конвертируются своими функциями `sbuild::addUint`, `sbuild::addInt`, `sbuild::addFloat`
- для base 10 в `addUint` используется собственный алгоритм деления на 10

Для `snprintf`:

- это универсальный форматтер
- он каждый раз парсит форматную строку
- поддержка разных спецификаторов делает его удобным, но добавляет накладные расходы
- для float на некоторых MCU поведение и цена по flash/RAM сильно зависят от libc и линковки

## Как проводить замер

Добавлен пример:

- `examples/StringPerfStudy/StringPerfStudy.ino`

Он сравнивает три сценария:

1. простой `badge`
2. средний по сложности `progress`
3. более тяжелый `hex-dump`, где есть цикл по байтам

Замеряется только форматирование строки. Вывод в `Serial` идет уже после бенчмарка, чтобы не смешивать время передачи и время сборки строки.

## Как интерпретировать результаты

### Если быстрее `snprintf`

Это нормально для короткой строки, особенно если она укладывается в один формат и не требует ручных циклов.

### Если быстрее `StringN`

Это особенно ожидаемо:

- при поэтапной сборке
- при hex-форматировании массивов байтов
- когда строка собирается из множества кусков
- когда важна предсказуемость без скрытых расходов libc

- плата: `ESP32-S3-DevKitC-1-N8R2`
- MCU: `ESP32-S3`, `240 MHz`
- среда: `PlatformIO`
- framework: `Arduino`
- build mode: `release`
- тестовый скетч: `examples/WidgetPerfConsole/WidgetPerfConsole.ino`

Что измерялось:

- только стоимость формирования строки команды
- без реальной передачи в `Serial`
- сравнение `TimberWidgets`/`StringN` против ручной сборки той же команды через `snprintf`

- для `TimberWidget` разумно оставить `StringN` основным способом сборки команд
- `snprintf` можно рассматривать только как точечную внутреннюю оптимизацию для отдельных простых табличных методов
- для frame/dump-виджетов `StringN` здесь явно предпочтительнее
## Практический вывод для TimberWidget

Для `TimberWidget` я бы ориентировался так:

- `StringN` лучше как основной рабочий инструмент библиотеки
- `snprintf` уместен для единичных очень простых строк
- для кадров, dump-виджетов, списков и составных команд `StringN` обычно логичнее и удобнее сопровождать

То есть вопрос здесь не только в сырой скорости CPU, но и в таких вещах:

- размер кода
- поведение на AVR
- отсутствие сюрпризов с `%f`
- предсказуемость буфера
- удобство пошаговой сборки сложных протокольных строк

## Что еще стоит проверить у себя

Если хочешь получить действительно честный ответ именно для своей платы, полезно сделать еще 3 отдельных замера:

1. AVR против ESP32
2. строка только с целыми числами против строки с float
3. Release-сборка с финальными флагами, которыми реально собирается проект
